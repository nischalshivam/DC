# PROJECT HANDOFF — Footage Collector (A‑to‑Z context for any future LLM)

> Give this file (and `01_TECHNICAL_REFERENCE.md`) to any AI assistant (Claude /
> ChatGPT / Gemini) and it will understand the entire project: what it is, why it
> was built this way, every design decision, the workflow, the limitations, and
> how to change or extend it. Read this fully before modifying anything.

---

## 1. WHAT THIS PROJECT IS (one paragraph)
A desktop tool that turns a **YouTube video‑essay script into ready‑to‑edit
footage**. You give it a beat‑by‑beat "visual instructor file" (generated by an
LLM from your clean narration script). For every beat it automatically downloads
a short **YouTube clip** (cropped to ~5s, normalised to 16:9 1080p) and several
high‑resolution **images**, plus **still frames grabbed from the clips**, and
organises everything into per‑scene folders with a `manifest.json`. It runs on
the user's own Windows PC (a simple GUI), uses **no paid APIs** (YouTube via
yt‑dlp, images via DuckDuckGo), and is designed for a **non‑technical user**.

## 2. THE USER & THEIR GOAL (the "need")
- The user makes faceless YouTube video essays (e.g. movie analysis like
  Scarface, The Thing). They are **non‑technical** — they cannot write code and
  need everything as turnkey as possible (one‑click GUI, setup scripts, guides).
- Core need: paste/provide a script → get **scene‑by‑scene clips + images** so an
  editor has a "visual buffet" per scene.
- Their stated priorities:
  - **Video clips are the crucial 60–70%** of the final video; images ~30%.
  - Everything must be **on‑topic and on‑point** (relevant to what the narration
    is talking about). Target ~**90% accuracy** (100% is impossible).
  - **No repeated** clips/images across scenes.
  - Images must be **high‑resolution, 16:9 landscape** (TV‑quality).
  - Prefer **no API keys / free**.
  - Works for **any topic** (movies, documentaries, WWE/sports, etc.).

## 3. THE FINAL SOLUTION (high level)
Two halves:

**(A) The "brain" — making the visual instructor file (outside this tool).**
An LLM reads the clean narration script and outputs a structured
**visual instructor file** (a `.txt`) that, for each beat, gives: the exact
narration, a CAPS scene name + concrete visual description, the exact on‑screen
**Spoken Line** (movie dialogue), **Image Search** terms, and optional
**Clip Links** (real YouTube URLs + timestamps). Helpers for this live in
`claude_project/` (Custom GPT / Gemini Gem / Genspark prompts + format spec).

**(B) The tool — turning that file into footage (this repo).**
`collector.py` parses the instructor file and, per beat:
1. tries any provided **Clip Links** (verifies they're real + relevant), else
2. **searches YouTube** (favouring real movie footage over reaction/breakdown
   videos), uses the **Spoken Line** to locate the exact moment via subtitles,
   downloads a ~5s section, normalises to 16:9 1080p;
3. extracts **still frames** from the clip (guaranteed on‑topic images);
4. downloads **landscape high‑res images** from DuckDuckGo;
5. writes everything to `output/scene_NNN/` + `manifest.json`,
   with **global de‑duplication** so nothing repeats.

## 4. THE JOURNEY (problem → fix) — WHY the code is the way it is
Read this so you don't "undo" hard‑won fixes.

1. **Started** as a keyless script→images+clips collector (search + crop).
2. **YouTube blocks cloud/datacenter IPs** ("Sign in to confirm you're not a
   bot"). → Tool is meant to run on the user's **own PC**; supports browser
   **cookies** / a **cookies.txt** file.
3. **Images were generic/irrelevant** (random "thing" images, anime, etc.).
   → (a) Always **anchor every query to the topic**; (b) **auto‑append the year**
   to the topic ("THE THING" → "THE THING 1982") to disambiguate; (c) filter to
   **landscape + high‑res** and rank by resolution + closeness to 16:9.
4. **Repeated images/clips across scenes.** → A shared **DedupState**
   (URL + content md5 + perceptual aHash) for images, and a global
   **used_sections** set for clips. Nothing repeats.
5. **Per‑2‑sentence chunking was weak.** → Added paragraph (word‑count) chunking,
   then moved to the **visual instructor file** approach for real relevance.
6. **The visual instructor file** (called "Type‑2"): the LLM does the semantic
   work; the tool just parses it. On‑Screen Text / Editor Notes are **ignored**
   for searching (kept only for a human editor).
7. **Clips were analysis/"Ending Explained"/reaction videos, not real footage.**
   → Title‑based relevance scoring: **penalise** explained/breakdown/reaction/
   review/essay; **prefer** "movie clip"/"scene"/HD/short videos.
8. **Editor‑style idea:** grab **still frames from the chosen clip** → always
   on‑topic images (`--frames-per-clip`).
9. **Windows breakages (very important):**
   - `yt-dlp` not on PATH → call it as **`python -m yt_dlp`**.
   - ffmpeg missing → bundle it in `bin/` (setup.bat downloads it) and resolve
     from there; pass `--ffmpeg-location`.
   - "Requested format is not available" (stale yt‑dlp) → **format fallback list**
     in `download_section` + **`update_tools.bat`** to update yt‑dlp.
   - Made failures **surface the real reason** instead of a silent "none".
10. **Cookies:** modern Chrome encrypts cookies → `--cookies-from-browser chrome`
    often fails on Windows. → Recommend a **cookies.txt** export (works reliably);
    GUI has a cookies.txt field.
11. **The big accuracy unlock — LLMs can't give real links/timestamps reliably**
    (they hallucinate). BUT they DO know exact **movie dialogue**. So we added the
    **`Spoken Line:`** field: the tool matches that exact dialogue against the
    video's **subtitles/transcript** to find the **precise timestamp** — no
    timestamp guessing needed. Web‑search AIs (Genspark / ChatGPT‑with‑search /
    Gemini) CAN supply real links; the tool **verifies** each link and **falls
    back to search** if it's bad. Result: **~90% accuracy** achieved.

## 5. END‑TO‑END WORKFLOW (how the user actually uses it)
1. Write/obtain the **clean narration script** (+ a title).
2. In a **Custom GPT / Gemini Gem / Genspark** (set up using files in
   `claude_project/`), paste the script → get the **visual instructor file**.
3. Save it as a `.txt`.
4. Open the tool (`run.bat` → GUI), pick the instructor file, set the **Topic**
   (e.g. "The Thing 1982"), choose a **cookies.txt**, click **Generate**.
5. Get `output/scene_NNN/` folders: `clip_*.mp4`, `shot*_*.jpg` (frames),
   `image_*.jpg`, `scene.txt`, plus `manifest.json`.

## 6. THE VISUAL INSTRUCTOR FILE = the contract between brain and tool
Per beat, these labels are parsed (case‑insensitive). Full spec:
`claude_project/FORMAT_SPEC.md`. Summary:
```
TITLE IN CAPS
Topic / context anchor: <Movie + Year>

SECTION HEADER (ALL CAPS)
Script Cue (narration): "<exact narration>"
Visual / Exact Clip to Use: <CAPS SCENE NAME>. "<short quote>". <concrete words>, about <Movie + Year>.
Spoken Line: <exact on-screen dialogue> | <another>      (KEY: finds exact timestamp)
Clip Links: <real youtube url ?t=sec>                    (optional; verified; never fabricate)
Image Search: <term> | <term> | <term>                   (recommended)
```
- READ for searching: `Script Cue`, `Visual / Exact Clip to Use`, `Spoken Line`,
  `Clip Links`, `Image Search`.
- IGNORED (human‑only): `On-Screen Text`, `Editor Notes`.

## 7. KEY DESIGN PRINCIPLES (do not regress these)
- **Always anchor queries to the topic + year.** Never produce an anchor‑less query.
- **Spoken Line (exact dialogue) → precise timestamp** via transcript phrase match.
- **Verify provided links; fall back to search.** Never trust an LLM link blindly.
- **Prefer real footage; penalise commentary/reaction.**
- **De‑duplicate globally** (images and clip sections).
- **Honesty over hype**: if footage doesn't exist, say so; don't fake it.
- **Non‑technical UX**: keep the GUI + one‑click .bat scripts working.

## 8. KNOWN LIMITATIONS / HONEST EXPECTATIONS
- Clips need footage that actually exists on YouTube; niche topics = lower hit rate.
- `Spoken Line` matching needs the candidate video to have a transcript/subtitles;
  pure‑action/silent beats rely on scene name + image search + frames.
- LLM‑provided links are often partly wrong (hallucinated) — the tool handles this.
- Long scripts (1‑hour docs) = many beats = long runtime; run in batches.
- Downloading copyrighted clips is the user's legal responsibility.

## 9. MAINTENANCE (long term)
- The tool's code is stable. The fragile dependency is **yt‑dlp ↔ YouTube**
  (YouTube changes every 1–2 months) and **ddgs ↔ DuckDuckGo**.
- Fix: run **`update_tools.bat`** (updates yt‑dlp + ddgs + requests + Pillow) and
  refresh **cookies.txt**. Details in `MAINTENANCE.md`.

## 10. HOW TO ASK AN LLM FOR A CHANGE LATER
Give the LLM BOTH handoff files, then say e.g.:
- "Add a `--vertical` mode that outputs 9:16 clips for Shorts."
- "Add support for an `Image Links:` line (direct image URLs)."
- "Make clips 8 seconds and grab 3 frames each."
Point it to `01_TECHNICAL_REFERENCE.md` §"Extension points" for where to edit.
Always test with `--dry-run` first, then a small `--max-scenes 3` run.
